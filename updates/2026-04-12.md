## 배포 전 필요한 설정

1. **Google Cloud Console**에서 OAuth 2.0 Web Client ID 생성
   - Authorized JavaScript origins: 프론트 도메인 + `http://localhost:5173`
   - Authorized redirect URIs: 불필요 (popup 방식)

2. **프론트엔드** `.env` 파일에 추가:
   ```
   VITE_GOOGLE_CLIENT_ID=<your-client-id>.apps.googleusercontent.com
   ```

3. **Terraform** 배포 시:
   ```
   TF_VAR_google_client_id=<your-client-id>.apps.googleusercontent.com terraform apply
   ```

4. 백엔드 빌드 후 `terraform apply`

## 커밋 메시지 작성 팁 (시니어 스타일)

- **왜(why)를 설명** — 무엇을 바꿨는지보다 왜 바꿨는지를 본문에 담는다
  - 나쁜 예: `fix login bug`
  - 좋은 예: `fix: persist auth session across page refreshes`
- **부작용(side effect)도 명시** — 변경으로 인해 같이 처리되는 동작을 설명
  - 예: "`exp` 검증으로 만료된 토큰이 자동으로 정리된다"
- **제목은 50자 이내, 본문은 72자 줄바꿈** — 터미널/GitHub에서 잘 읽힘
- **Conventional Commits 형식 사용** — `feat:`, `fix:`, `refactor:`, `chore:`, `docs:` 등
  - 자동 changelog 생성 도구와 호환됨
- **커밋 하나 = 하나의 논리적 변경** — 리뷰어가 diff 읽을 때 맥락이 명확해짐
  - 여러 버그를 고쳤다면 커밋도 나눠서

## Commit Message Tips (Senior Dev Style)

- **Explain the why, not just the what** — the body should answer "why was this necessary?"
  - Bad: `fix login bug`
  - Good: `fix: persist auth session across page refreshes`
- **Call out side effects** — mention behaviour that changes as a consequence
  - e.g. "expired tokens are evicted automatically via `exp` validation"
- **Subject ≤ 50 chars, body wrapped at 72** — readable in terminals and GitHub
- **Follow Conventional Commits** — `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`
  - Enables automated changelog generation
- **One commit = one logical change** — makes reviews and bisects easier
  - If you fixed multiple bugs, split them into separate commits
